## PowerShell / Windows

- `bash_tool` НЕ работает для команд на сервере — только через srv.ps1/home.ps1
- `-SkipCertificateCheck` не работает в PowerShell 5 → заменять на `[System.Net.ServicePointManager]::ServerCertificateValidationCallback = { $true }`
- `srv.ps1` ломает сложные команды с кавычками и кириллицей → большие файлы через GitHub → curl

- Telegram MCP: `my.telegram.org/apps` может отдавать безликий `ERROR` при создании app, даже если форма заполнена корректно. Рабочий обход на домашнем ПК: публичные credentials Telegram Desktop + импорт локальной Telegram Desktop `tdata` в `mcp-telegram`.
- Telegram MCP / GramJS на Windows: `StoreSession` ломается на абсолютных путях и на forward slash (`.telegram-agent/sessions/...`) — появляются нерабочие файлы с `%2F`. - Telegram MCP stdio: `.cmd`-обёртка может зависать в smoke-тестах MCP SDK. Надёжнее прямой `node.exe scripts/telegram-mcp/mcp-telegram-wrapper.mjs`; wrapper сам делает `chdir(%USERPROFILE%)` и импортирует установленный `mcp-telegram`.
- Telegram массовые подписки через GramJS: после нескольких десятков `channels.JoinChannel` Telegram начинает отдавать `FLOOD`/`FLOOD_WAIT`. Не давить цикл подряд; подписывать партиями с паузами и `--baseline-new`, чтобы новые каналы не залили старую историю в дайджест.

## WireGuard (домашний сервер ↔ VPS)

- **ГЛАВНОЕ:** домашний сервер за NAT — ListenPort в конфиге НЕ влияет на внешний порт. Роутер подменяет порт на случайный. WireGuard обновляет endpoint автоматически по входящим пакетам — это штатное поведение.
- **Причина потери туннеля после перезагрузки:** PublicKey пира в wg0.conf домашнего сервера указывал на wg-easy (порт 51820), а не на wg1 (порт 51822) — разные ключи, handshake невозможен криптографически.
- Правильный PublicKey VPS для wg1: `QTSKLBhGyvL+5Sm4EZbnu6m0iL2ufu+8nD3rrIUt00Q=`
- Правильный Endpoint в wg0.conf домашнего: `147.45.238.120:51822`
- На VPS в wg1.conf — НЕ прописывать endpoint для домашнего сервера, WireGuard выучит сам
- `sudo systemctl enable wg-quick@wg1` на VPS — через `ln -s` т.к. `systemctl enable` возвращает stderr и shell-api даёт 500
- Доступ к домашнему серверу — ТОЛЬКО через `home.ps1`, НЕ через `srv.ps1`. Маршрут: nginx VPS :7724 → 10.8.0.27:7722

## Nginx конфиги через srv.ps1

- `printf`, `python3 -c`, heredoc — всё ломает `$host`, `$http_upgrade` при передаче через srv.ps1

То есть дело не в условии (`InterfaceAlias` vs `LocalAddress`) вообще.
  - **РЕАЛЬНАЯ ПРИЧИНА**: в Windows Filtering Platform / `New-NetFirewallRule`, когда для ОДНОЙ И ТОЙ ЖЕ программы (`-Program`) существуют и Allow-, и Block-правило, которые оба потенциально применимы — **Block побеждает Allow всегда, независимо от того, насколько специфичны условия Allow** (это документированное поведение Windows Firewall, не баг конкретного условия). Поэтому модель "Allow только через VPN + Block всё остальное" для одного и того же exe **в принципе не может работать** через стандартные `New-NetFirewallRule`/`netsh advfirewall` — нет способа задать вес/приоритет правила через эти инструменты, чтобы специфичный Allow забил общий Block (это умеют только официальные VPN-killswitch реализации через прямой WFP API с кастомными sublayers/weight, не доступно из PowerShell-cmdlets).
  - **ПРАВИЛЬНАЯ РАБОЧАЯ АРХИТЕКТУРА (подтверждена пользователем, "воркс")**: убрать вообще Allow-правила и общий Block. Вместо этого — **только Block, привязанный по `-LocalAddress` к подсети НЕ-VPN адаптера** (в данном случае Ethernet `192.168.1.0/24`, узнать через `Get-NetIPAddress -InterfaceAlias Ethernet`). Логика: дефолтная исходящая политика Windows Firewall — Allow (`Get-NetFirewallProfile` показывает `DefaultOutboundAction: NotConfigured` = Allow по умолчанию), поэтому весь VPN-трафик (sourced из `10.8.1.3`/`fd58:baa6:dead::1`, вне диапазона `192.168.1.0/24`) просто не попадает под Block и идёт под дефолтным Allow — никакого конфликта Allow/Block для одного exe не возникает, потому что Allow-правила вообще нет. Если VPN упадёт целиком — ОС начнёт роутить через Ethernet, сокет получит local address из `192.168.1.0/24`, и Block сработает как задумано. Финальные правила: `Claude killswitch - block non-VPN (claude.exe)` и `(Claude.exe)`, по одному на exe, `-Direction Outbound -Action Block -LocalAddress "192.168.1.0/24"`.
  - **Вывод для повторения на другом компьютере**: НЕ пытаться делать "Allow через интерфейс/IP VPN + Block остальное" для одного процесса — гарантированно сломается из-за Block>Allow precedence. Сразу делать "Block по подсети LAN-адаптера(ов), без Allow вообще". Подсеть LAN-адаптера узнавать через `Get-NetIPAddress -InterfaceAlias <Ethernet/WiFi>`.

## pyRevit на рабочем ПК — установка и загадочный баг с длинным путём (2026-07-01)

- **Инсталлятор pyRevit скачивался повреждённым дважды подряд** через `Invoke-WebRequest`/`WebClient` в связке с MCP-таймаутом инструмента (~60-90с обрывал скачивание файла ~87 МБ на медленном канале рабочей сети, файл оставался частично записанным, но без ошибки). **Фикс**: качать через `Start-BitsTransfer -Asynchronous` (фоновая передача, не привязана к таймауту вызывающего инструмента) + опрашивать `Get-BitsTransfer` в цикле + `Complete-BitsTransfer` в конце. Обязательно проверять размер/хэш файла (`Get-FileHash`) против размера ассета в GitHub API (`api.github.com/repos/.../releases/latest`) перед запуском инсталлятора — иначе получаешь малопонятную ошибку "setup files are corrupted".
- **GUI-инсталлятор pyRevit (`pyRevit_*_signed.exe`) не поддерживает тихую установку** — `/quiet`, `/S`, `/silent`, `/VERYSILENT` не распознаются и либо тихо падают (exit 1), либо инсталлятор всё равно требует GUI. Через `pyrevit CLI` (`pyrevit clone <name>` без `--source`) тоже не получилось — команда виснет намертво на этой сети без прогресса (внутренний LibGit2Sharp-клиент, видимо, не совместим с чем-то на этой сети/прокси) — путь пришлось пройти руками через GUI wizard.

URL: `https://aka.ms/dotnet/8.0/windowsdesktop-runtime-win-x64.exe`.
- **git-клон pyRevit (`git clone` полный, без `--depth`) падает на Windows с "Filename too long"** — у pyRevit в истории есть файлы с очень длинными вложенными путями (`extensions/pyRevitBundlesCreatorExtension.extension/pyRevit Bundles Creator.tab/...`).
