# ПРАВИЛА НАПИСАНИЯ КОДА

> Клод обязан следовать этим правилам при написании любого кода для Сергея.

---

## Принцип 1 — Сначала архитектура, потом код

Перед написанием нетривиального кода (новый сервис, >2 файлов, или задача неясна):

1. Клод переформулирует задачу своими словами — убедиться что понял правильно
2. Предлагает структуру: какие файлы, модули, что в каждом
3. Описывает поток данных: откуда приходит → что происходит → куда уходит
4. Объясняет выбор технологий — почему именно это, а не другое
5. Спрашивает «Всё верно? Начинаем?» и ждёт ответа

**Делать сразу без обсуждения:**
- Правка одного файла / одной функции
- Исправление явного бага
- Добавление конфига к готовому сервису

**Почему это важно:** правильная архитектура до написания экономит часы переделок. Один файл на 3000 строк невозможно читать и поддерживать.

## Принцип 2 — Комментарии обязательны

Каждая нетривиальная функция имеет комментарий.

**Когда комментарий нужен — над функцией, одна строка:**
```python
# Конвертируем мм → футы: Revit хранит координаты во внутренних единицах (футы)
def mm_to_feet(mm: float) -> float:
    return mm / 304.8

# pdfminer вместо PyMuPDF — у PyMuPDF кракозябры для кириллицы (Identity encoding)
def extract_text(pdf_path: str) -> str:
    ...

# BM25 + fastembed + RRF даёт лучше результат чем только векторный поиск
def hybrid_search(query: str) -> list:
    ...
```

**Когда комментарий не нужен** — если имя функции уже всё объясняет:
```python
def get_user_by_id(user_id: int): ...
def format_date(date) -> str: ...
```

**В конфигах и docker-compose** — комментировать неочевидные параметры:
```yaml
# client_max_body_size для загрузки фото Immich (до 500MB)
client_max_body_size: 50000m;
```

## Принцип 3 — Декомпозиция файлов

Один файл = одна ответственность.

| Размер файла | Что делать |
|---|---|
| < 200 строк | Норм |
| 200–400 строк | Подумать о разделении |
| > 400 строк | Обязательно разделить |

**Типовая структура Python-проекта:**
```
project/
├── main.py        ← точка входа, минимум логики
├── config.py      ← все константы и env-переменные
├── db.py          ← работа с БД
├── api.py         ← внешние HTTP запросы
└── utils.py       ← вспомогательные функции
```

**Типовая структура FastAPI сервиса:**
```
service/
├── main.py        ← запуск, регистрация роутеров
├── config.py      ← переменные окружения (pydantic-settings)
├── models.py      ← Pydantic модели запросов/ответов
├── db.py          ← подключение к БД, запросы
└── routes/
    ├── health.py  ← /health endpoint
    └── data.py    ← основная логика
```

## Принцип 4 — Именование

| Что | Стиль | Пример |
|---|---|---|
| Python файлы, папки | `snake_case` | `parse_health_db.py` |
| Конфиги, YAML, nginx | `kebab-case` | `docker-compose.yml` |
| Функции | глагол + объект | `get_user`, `create_grid`, `parse_pdf` |
| Константы | `UPPER_SNAKE_CASE` | `DB_HOST`, `API_KEY` |
| Классы | `PascalCase` | `GridBuilder`, `HealthSync` |
| Переменные | говорящие, не `x`/`tmp`/`data` | `user_id`, `pdf_path` |

## Принцип 5 — Команды запуска всегда в context.md

Раздел **ТЕХНИЧЕСКИЕ ДЕТАЛИ** каждого проекта обязан содержать:
```
# Запуск локально
python main.py

# Деплой на сервер
docker compose up -d

# Проверка что работает
curl http://localhost:8080/health

# Перезапуск
systemctl restart service-name
```

Без этого через месяц невозможно вспомнить как запустить.

---

## Принцип 6 — Секреты не в коде

Никаких паролей, токенов, API ключей прямо в коде.
- Python: `os.environ.get('API_KEY')` + `.env` файл (в .gitignore)
- Docker: `.env` файл рядом с docker-compose.yml (в .gitignore)
- Nginx: токены в конфиге сервиса, не в nginx.conf
