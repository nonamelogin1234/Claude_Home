# DECISIONS.md — DocAI

## Ключевые решения

### Библиотека для извлечения текста: pdfminer.six ✅

**Проблема:** PyMuPDF (fitz) возвращает кракозябры для кириллицы из-за Identity encoding.
Функция `is_garbled()` на стандартном тексте не помогает — мусор выглядит как валидный Unicode.

**Решение:** pdfminer.six правильно читает ToUnicode CMap из шрифтов и корректно декодирует
кириллицу. Проверено на test.pdf (14 стр.) — все страницы читаются чисто.

**Подтверждение:** PyMuPDF возвращает `ord=65533` (U+FFFD) для кириллических символов,
тогда как pdfminer возвращает правильные коды (напр., 1048=И, 1055=П, и т.д.).

### Детекция мусорного текста

Функция `is_garbled()`:
- Считает символы замены `U+FFFD` (более 3 — мусор)
- Проверяет что среди букв есть кириллица или латиница
- Если букв много, но нет ни кириллицы ни латиницы → мусор

**Важно:** Простая проверка на «выглядит ли как ASCII» не работает — у Windows
терминал отображает cp1251/cp866, поэтому в консоли видны кракозябры, но в файле
(при записи с encoding='utf-8') текст корректный.

### OCR: Tesseract (опционально)

На данной машине Tesseract не установлен. Код поддерживает fallback через:
- `pytesseract` + PyMuPDF для рендеринга страниц в изображения (300 DPI)
- Язык: `rus+eng`
- Активируется только для страниц с мусором/пустым текстом

### Структура JSON

```json
{
  "docai_version": "1.0",
  "source_file": "...",
  "total_pages": N,
  "pages": [{ ... }],
  "all_numbers": [{ "page": N, "value": "...", "type": "...", "context": "..." }],
  "cross_page_values": { "значение": [{ "page": N, "context": "..." }] },
  "stats": { ... }
}
```

`cross_page_values` — значения встречающиеся на 2+ страницах → это то что Claude проверяет
на противоречия.

### Паттерны числовых значений

Детектируются типы:
- `отметка` — `±N,NNN` или `0,000`
- `размер` — `NxM` с разделителями `х/×/x`
- `площадь` — `N,NN м2/м²/кв.м`
- `объём` — `N,N м3/м³`
- `нагрузка` — `N кН/м², кг/м², МПа` и т.д.
- `размер_м` — `N,NN м` (высоты, длины)
- `количество` — `N шт/п.м/рул`
- `число` — общий fallback

**Особенность:** Российская документация использует запятую как десятичный разделитель
(`12,73 м`), иногда точку (`3.17 м`). Паттерны учитывают оба варианта.

## Грабли

### Windows консоль и UTF-8

При запуске в Windows CMD/PowerShell — кириллица в stdout отображается как кракозябры
(консоль использует cp1251/cp866). Это **не** ошибка программы — данные в JSON/файлах
корректные. Вывод прогресса в терминале будет нечитаемым, но JSON всегда в UTF-8.

Для правильного вывода в консоль нужно: `chcp 65001` перед запуском.

### Чертёжные страницы (векторные)

На страницах с планами/фасадами (стр. 11-14 test.pdf) pdfminer извлекает каждый символ
как отдельный текстовый блок. Числа дробятся: `1 6 5 0` вместо `1650`.
Это ограничение pdfminer при работе с векторными чертёжными шрифтами.

Числовые паттерны (4+ цифры) частично компенсируют это, но некоторые значения теряются.
Для полного решения нужен OCR чертёжных страниц (Tesseract).

### pdfminer и Identity-H без ToUnicode

Если шрифт не имеет ToUnicode CMap — pdfminer тоже даст мусор. На данном PDF это
не проблема, но для других документов нужно проверять и переключаться на OCR.

## DocAI v3 — Desktop App (app.py)

### UI: PyQt6

PyQt6 уже установлен в среде. Лучший вариант для Windows десктопа:
- Нет зависимости от браузера или внешнего сервера
- Полноценные нативные диалоги (QFileDialog, QInputDialog, QMessageBox)
- QThread для фоновой обработки без заморозки UI

### База данных: SQLite (db.py)

Путь: `%APPDATA%\DocAI\docai.db`
Таблицы: `projects`, `files`, `settings`
CASCADE DELETE: удаление проекта удаляет все его файлы из БД.

Данные файлов (JSON + индекс) хранятся в `%APPDATA%\DocAI\data\{file_id}\`:
- `extracted.json` — результат docai.py
- `extracted.vecs.npy` — векторы
- `extracted.meta.json` — метаданные чанков

Это позволяет перемещать PDF на диске, не теряя индекс (если путь к PDF менялся — просто пересчитать хеш).

### Кэш модели embeddings

`FASTEMBED_CACHE_PATH = %APPDATA%\DocAI\models`
Устанавливается через `os.environ` до любых `import fastembed`.
Модель скачивается один раз при первой обработке файла.

### Многофайловый поиск: _multi_search()

`ask.search()` создаёт новый `TextEmbedding` при каждом вызове.
Для поиска по нескольким файлам создан `_multi_search()` в app.py:
- Создаёт embedder один раз
- Обходит все indexed JSON файлы
- Объединяет все чанки и применяет единый RRF

### Фоновая обработка: QThread

`ProcessWorker(QThread)`:
1. Перехватывает stdout (StringIO) из docai и ask
2. Эмитит `progress(str)` сигналы в UI
3. По завершении: `finished(bool, str)`

`AskWorker(QThread)`:
- Вызывает `_multi_search()` + `ask_claude()`
- API key берётся из DB settings, выставляется в `os.environ`

### Изменение файлов: детекция по MD5

`db.file_needs_reindex(file_id)`:
- Если статус != 'ready' → True
- Иначе считает MD5 первых 2MB PDF и сравнивает с хранимым хешем

### PyInstaller: build.py

`--onefile --windowed` — единый exe без консоли.
Модель embeddings не упаковывается — скачивается при первом запуске.
Data файлы (docai.py, ask.py, db.py) добавляются через `--add-data`.

### Архитектура файлов

```
app.py       -- PyQt6 GUI + многофайловый поиск
db.py        -- SQLite CRUD
docai.py     -- PDF → JSON (без изменений)
ask.py       -- RAG: build_index, search, ask_claude (без изменений)
build.py     -- PyInstaller сборка
```

## DocAI v2 — RAG (ask.py)

### Библиотека embeddings: fastembed + paraphrase-multilingual-MiniLM-L12-v2

**Выбор:** `fastembed` (onnxruntime-based) вместо full sentence-transformers + PyTorch.
- Не требует PyTorch (~1 GB) — только onnxruntime (~10 MB)
- Модель `paraphrase-multilingual-MiniLM-L12-v2` — 0.22 GB, поддерживает русский
- Скачивается автоматически при первом запуске в `%TEMP%/fastembed_cache`

### Хранилище индекса: два файла рядом с JSON

- `<stem>.vecs.npy` — numpy array (N, 384) float32
- `<stem>.meta.json` — метаданные чанков + имя модели

Без FAISS/ChromaDB — простой numpy cosine similarity. Для 200 чанков это мгновенно.

### Гибридный поиск: Semantic + BM25 через RRF

**Проблема:** чистый семантический поиск даёт слабый recall для точных числовых запросов.
Пример: "площадь застройки" → чанк с "161,9 м2" попадает на 12-е место из-за того,
что начало чанка ("вной пожарной опасности...") не семантически близко к запросу.

**Решение:** Hybrid search = Semantic + BM25 (rank_bm25), объединённые через
Reciprocal Rank Fusion (RRF, k=60).

Результат: "площадь застройки" → чанки с 161,9 м2 и 126,8 м2 на 1-2 месте.

### Чанкинг

1. Текстовые блоки от pdfminer → основа чанков
2. Короткие блоки (<80 симв.) мержатся с предыдущим
3. Длинные (>600 симв.) разбиваются по предложениям
4. **Числовые чанки** — специальные чанки из `all_numbers` (отметки, площади, размеры):
   формат `[Стр. N] [тип: значение] контекст`. Улучшают recall для числовых запросов.

### Claude API (ask.py)

- Модель: `claude-sonnet-4-6`
- System prompt: роль эксперта по проектной документации
- Передаётся TOP_K=6 релевантных чанков
- Ключ: `ANTHROPIC_API_KEY` из переменной окружения

**Если ключ не установлен:** скрипт выводит инструкцию `set ANTHROPIC_API_KEY=sk-ant-...`

### Команды

```
# Один раз — построить индекс:
python ask.py --build test_output.json

# Вопрос к документации:
python ask.py "какая высота этажа?"

# Без Claude (только поиск):
python ask.py --no-llm "площадь застройки"

# Указать конкретный JSON:
python ask.py --json test_output.json "отметка кровли"
```
